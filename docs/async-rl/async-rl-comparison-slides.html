生成完成即流式送入训练，无需等 long-tail → 消除空泡。代价：训练数据可能来自旧权重 → staleness

核心 trade-off：消除 long-tail 空泡、提高 GPU 利用率 vs. 训练数据与当前策略的偏移（off-policy）。所有 async RL 的设计选择都围绕这个 trade-off 展开。

① Rollout Buffer

生成数据如何暂存？

RL step 的起点：推理引擎生成 rollout，数据进入 buffer 等待训练消费

三种 Buffer 策略

SLIME: Double-Buffer

1
Buffer 深度
  • 主循环亲自调度 rollout
  • 最多 1 个 batch in-flight
  • Gen N+1 与 Train N 并行
  • 调度逻辑 ~40 行

NeMo-RL: Replay Buffer

1~N
Buffer 深度 (可配)
  • 后台 Collector 持续生成
  • 训练从 Buffer 按需取数据
  • Target weight version 匹配
  • Stall 语义：不够就等

verl: Bounded Queue

N
Queue 深度 (由 staleness 决定)
  • 独立 Rollouter actor streaming
  • 逐 sample 粒度推入 queue
  • 满了丢老数据，不阻塞 producer
  • 双层 backpressure 控制
关键区别：SLIME 的 buffer 深度固定为 1（最多 1 步 staleness）；NeMo-RL 通过 max_trajectory_age_steps 配置深度；verl 通过 staleness_threshold 间接控制 queue 容量。

② Partial Rollout

生成被打断怎么办？

Rollout 进行中遇到权重同步或过采样完成 → 还在飞的请求何去何从？

Partial Rollout：三种策略

NeMo-RL: 不支持

  • 所有 sample 必须完整运行
  • 没有 abort / recycle 机制
  • In-flight 模式不中止正在生成的请求

SLIME: Abort + Recycle

  • 凑够train batch就 abort 剩余请求
  • 被 aborted 请求的已生成 token 放入回收 buffer
  • 下次优先从 buffer 取 partial sample
  • 旧 token 的 loss_mask=0
优势：不浪费已完成的工作 + 可利用 prefix cache

verl: Abort + Prefix Continue

  • Hard abort engines后，每个被 abort 的 request 循环重试
  • 输入prompt += 已生成 tokens，继续生成
  • 对上层调用者完全透明，感知不到被 abort 过
代价：KV cache 被清空，无法利用 prefix cache

时序对比：生成被打断时

第二层：样本进来后，loss 怎么修？

这时才比较 TIS / OPSM / mask-IS / seq-mask-TIS。它们解决的是“样本已经进 trainer 之后”的 off-policy 偏移。

Token-level 修正

策略公式框架
TIS clamp(πθ/πold, max=T) SLIME NeMo-RL verl
Mask-IS / Rejection ratio 超限 → 置零或 mask NeMo-RL verl

Sequence-level 修正

策略Seq metric触发条件框架
OPSM seq_kl = mean(log π_old - log π_cur) adv < 0 且 seq_kl > δ → mask SLIME
seq-mask-TIS geo-mean IS = exp(mean(log π_old - log π_rollout)) geo-mean IS 不在 [min,max] → mask NeMo-RL
Rejection seq_sum/mean/max of k1/k2/k3(log-ratio) sequence divergence statistic 超阈值 → mask verl
三者的关联：它们都从 token-level 的 log-prob difference / log-ratio 出发，先聚合成一个 sequence scalar 再决定是否 mask。
SLIME 保留在 log-space；NeMo-RL 对同类量再取 exp，变成几何均值 IS ratio；verl 则把 log-ratio 变成 k1/k2/k3 这类 divergence statistic 再做 rejection。

④ 权重同步协议

新权重如何推送回推理引擎？

训练完成 → 新权重需要同步到推理 GPU → 这是一个 RL step 的最后一环

权重同步：三种中断策略

SLIME: Drain → Sync

① 外层调度先 drain in-flight rollout
② pause_generation
③ flush KV cache
④ TP all-gather
⑤ Megatron → HF 转换
⑥ Bucketed NCCL broadcast
⑦ continue_generation

需要 HF 格式转换 + 锁协调

NeMo-RL: Pause → Sync

① 暂停新生成
② (Blocking) 等 pending 完成
   (In-flight) 跳过等待
③ NCCL collective broadcast
   via vLLM collective_rpc
④ (可选) invalidate KV cache
⑤ version++ + resume generation

无需格式转换，vLLM 直接接收

verl: Abort → Sync

① Hard abort 所有请求
② Sleep replicas (释放 KV)
③ 建立 NCCL 通信组
④ Bucketed NCCL broadcast
⑤ 销毁通信组
⑥ Wake replicas (恢复引擎 / 重新占用 KV 显存)
⑦ Resume generation

显存借用：sleep 腾 KV 空间放 buffer

NeMo-RL 的 In-flight Weight Update 是独有的激进特性：权重同步时不等待正在进行的生成完成，允许部分 token 用旧权重、部分用新权重生成。这依赖 vLLM V1 的 collective_rpc 能力。可选配合 KV cache invalidation 来缓解不一致。

Key Takeaways

1. 三种设计哲学，一个核心 trade-off

SLIME 选择简洁（double-buffer + 40行主循环）；NeMo-RL 选择激进（in-flight weight update + 深 replay buffer）；verl 选择通用（完全解耦的独立 actor 架构）。但本质上都在平衡 GPU 利用率 vs. off-policy 偏移。

2. Buffer 深度是关键 knob

SLIME 固定为 1 步（最安全）；NeMo-RL 和 verl 允许多步（更高 throughput）。NeMo-RL 的官方配置已有 max_trajectory_age_steps=8 的实例，配合 MoE 24 节点场景。Buffer 越深，版本 filtering / staleness handling 就越关键，而不只是 loss 层 correction。

3. Staleness 修正没有银弹

三框架都用了 IS correction，但真正的差异在于 loss 前是否还有 filtering / selection：SLIME 默认主路径以 depth=1 约束 lag，再做 TIS/OPSM；NeMo-RL 用 target-match + age-window 决定哪些样本能进当前步，再做 IS；verl 则把 queue threshold、rejection 和 MIS 组合得更系统。哪种更好取决于 staleness 程度和系统复杂度预算。

4. 权重同步是最大的工程瓶颈

三框架在权重同步上的差异最大：SLIME 需要格式转换 + 锁协调；NeMo-RL 利用 vLLM 的 collective_rpc 最简洁；verl 的 sleep/wake 显存借用最灵活。中断策略（drain/pause/abort）直接决定 partial rollout 的处理方式。
